What It Is

edvat is a Go command-line migration planner for PostgreSQL projects using Atlas HCL. Atlas OSS already knows how to parse, diff, and plan core PostgreSQL schema objects such as schemas, tables, columns, constraints, indexes, and enums. edvat adds planning support for PostgreSQL object families that are not delegated to Atlas in this repository, such as extensions, functions, policies, permissions, triggers, views, domains, sequences, roles, users, and seed data.

The output is an Atlas-style migration directory: timestamped .up.sql files, optional .down.sql files when statements have reverse SQL, and an atlas.sum checksum file.

Commands

edvat capabilities

Prints every object family, who owns it, and whether support is delegated, experimental, or unsupported.

edvat migrate diff <name>

Builds a migration plan and writes a timestamped <name>.up.sql file into the configured migration directory.

edvat migrate hash

Recomputes the Atlas checksum file for the configured migration directory.

edvat migrate diff add_billing --env local --config atlas.hcl

Project Config

internal/project.LoadEnv reads an Atlas HCL config file and selects one env. The env supplies the schema source files, the migration directory, and optional seed settings.

data "hcl_schema" "app" {
  paths = ["schema.pg.hcl", "security.pg.hcl"]
}

env "local" {
  schema { src = data.hcl_schema.app.url }
  migration { dir = "migrations" }

  data {
    mode = UPSERT
    src = ["seed.sql"]
  }
}

Supported schema source forms are a string path, a list of paths, or data.hcl_schema.<name>.url. Paths are normalized relative to the config file directory. Seed mode defaults to INSERT when no mode is configured.

Planning Pipeline

The central workflow is migrationplan.Build. It creates a desired state from the configured files, optionally creates a current state from a dev database, produces SQL from several sources, and then orders and validates the result.

CLI migrate diff
  -> project.LoadEnv(config, env)
  -> migrationplan.Build(cfg, options)
     -> loadDesiredMigrationState
        -> baseatlas.LoadDesired     (Atlas-owned schema)
        -> pgext.Parse*Files         (edvat-owned object families)
     -> optionally migratedir.ReplayURL
     -> optionally inspectCurrentMigrationState
        -> baseatlas.InspectCurrent  (Atlas-owned current schema)
        -> pgext.Inspect*URL         (edvat-owned current state)
     -> baseatlas.Diff + baseatlas.PlanSQL
     -> appendPgExtStatements
     -> loadSeedStatements
     -> suppressManagedExclusionConstraintDrops
     -> orderMigrationStatements
     -> destructiveFindings gate
  -> migratedir.Writer.Write
     -> write .up.sql, optional .down.sql, atlas.sum

The Important Split

baseatlas.LoadDesired deliberately strips edvat-owned HCL blocks before handing source text to Atlas. That means a single schema file can contain normal Atlas objects and edvat-only blocks. Atlas sees what it owns; edvat separately parses the blocks it owns.

Modes And Flags

Mode or FlagEffect
No --dev-urlPlans from an empty current state. This is useful for initial migrations or generating all desired objects from scratch.
--dev-urlInspects a live PostgreSQL database before diffing. Atlas-owned schema and edvat-owned object families are both inspected.
--replay-migrationsRequires --dev-url. Applies existing migration SQL files from the migration directory to the dev database before inspection, so the diff starts from replayed migration history.
--manage-rolesEnables role diffing and DDL. Desired role blocks are rejected without this flag because role changes are security-sensitive.
--manage-usersEnables user diffing and DDL. Desired user blocks are rejected without this flag. Passwords are unsupported/rejected by the user-family implementation.
--allow-destructiveAllows a plan containing statements marked destructive. Without it, destructive findings stop the command and tell you to rerun after review.
Seed INSERTGenerates INSERT ... ON CONFLICT (...) DO NOTHING for each desired row.
Seed UPSERTGenerates INSERT ... ON CONFLICT (...) DO UPDATE SET ... where non-key columns can change.
Seed SYNCWithout current-row inspection, emits upserts plus a delete for desired keys not present. With --dev-url, inspects current rows and emits targeted inserts, updates, and deletes.

Object Families

The capability model has two owners. atlas_oss families are delegated to Atlas. edvat families are parsed, inspected, diffed, and planned by code in internal/pgext or internal/seed.

Most pgext files follow this shape: define a state struct, parse HCL blocks from schema source files, optionally inspect current PostgreSQL catalogs, diff current versus desired maps, and return baseatlas.Statement values.

type Family struct { ... }

func ParseFamilyFiles(paths []string) (FamilyState, error)
func InspectFamiliesURL(ctx context.Context, url string) (FamilyState, error)
func DiffFamilies(current, desired FamilyState) []baseatlas.Statement

Seed Data

Seed data comes from two places. HCL data blocks can be embedded in schema source files. Raw SQL seed files can be listed in the env data.src setting.

data {
  table = schema.public.countries
  key = [code]
  rows = [
    { code = "US", name = "United States" },
    { code = "SE", name = "Sweden" },
  ]
}

If a data block omits key, edvat tries to derive key columns from the desired Atlas realm primary key. If it cannot find key columns, seed SQL generation fails because conflict handling and synchronization need stable identity.

SQL seed files are split on semicolons while respecting single-quoted strings and $$ dollar-quoted blocks. Each split statement is appended as a seed statement.

Ordering And Safety

migrationplan.orderMigrationStatements applies a stable ordering to reduce dependency problems. The rough order is schemas, extensions, types/domains/collations/sequences, tables, alters, indexes, casts, routines, aggregates, foreign-data objects, triggers, policies/grants, views, seed data, then drops near the end.

There are extra passes for dependency-sensitive cases:

  • Tables are topologically ordered by REFERENCES clauses where possible.
  • Functions can be moved before tables that reference them.
  • Created tables, functions, and views can be reordered when same-order objects reference each other.
  • Atlas-generated drops for exclusion constraints managed by edvat are suppressed when the desired edvat exclusion exists.

Destructive statements are detected after ordering. If any destructive findings exist, the command fails unless --allow-destructive is present. This is a review gate, not a SQL parser.

Tests

The repository is heavily test-oriented. Almost every object-family file has a matching _test.go. There are also CLI tests, migration directory tests, seed tests, capability tests, and integration tests using PostgreSQL testcontainers.

go test ./...

When changing object-family behavior, start with the matching file in internal/pgext and its test. When changing orchestration or ordering, start in internal/migrationplan.

How To Read The Code

  1. Start at cmd/edvat/main.go to understand the public CLI and flags.
  2. Read internal/project/project.go to understand how atlas.hcl is decoded into EnvConfig.
  3. Read internal/migrationplan/plan.go for the orchestration sequence.
  4. Read internal/migrationplan/state.go to see every desired/current state family and where diffs are appended.
  5. Read internal/baseatlas/baseatlas.go to understand the Atlas boundary and edvat-block stripping.
  6. Read one simple pgext family, such as extension.go, then one routine family, such as function.go, to learn the common parse/inspect/diff pattern.
  7. Read internal/seed/seed.go for the seed modes and row-diff logic.
  8. Read internal/migrationplan/dependencies.go if generated statement order surprises you.
